# U4Claude v1.3.0

In-app updates via Sparkle. The custom GitHub-API poller is gone; Sparkle now owns the entire update flow — background check, EdDSA signature verification, download, and install.

## What changes for users

- **Updates install with one click instead of three.** Previously: get notified, click "Download", download DMG to `~/Downloads`, drag U4Claude.app onto Applications, replace, relaunch. Now: get the standard Sparkle "An update is available" sheet, click "Install Update", and the app relaunches into the new version automatically.
- **Updates are signature-verified.** The DMG contents are validated against the EdDSA public key embedded in the app at build time. A tampered or corrupted update is rejected before install.
- **Background check still 24-hour cadence**, same as before.
- **No menu bar badge for "update available"** — Sparkle's own modal is the source of truth. The badge UI in the codebase is dormant for now (will be cleaned up in a follow-up).

## What changes for the maintainer

- **EdDSA keypair generated.** Public key embedded in the app via `INFOPLIST_KEY_SUPublicEDKey` build setting; private key lives in your login keychain (managed by Sparkle's `generate_keys` tool, never copy it elsewhere). Future releases must be signed with the same key — losing it means every existing install becomes orphaned and can't auto-update without a manual reinstall.
- **`appcast.xml` at the repo root** drives the update feed. Hosted via raw GitHub: `https://raw.githubusercontent.com/arcanii/Usage4Claude-Arcanii/main/appcast.xml`. Each release adds a new `<item>` entry with `sparkle:edSignature` (from `sign_update`) and `length` (from the DMG byte size).
- **`scripts/build.sh` runs `sign_update` after notarize+staple** and prints the appcast `<enclosure>` line ready to paste into `appcast.xml`. The build step is skipped (with a hint) if `SIGN_UPDATE` env var doesn't point at the Sparkle binary.

## Migration path

- **Existing v1.2.0 installs upgrade via the OLD update mechanism one last time.** They have the GitHub-API poller; they'll detect v1.3.0 as a published GitHub Release (assuming the user publishes one with the DMG) and prompt to download. After upgrading to v1.3.0, Sparkle takes over for all subsequent versions.
- **v1.3.0 onwards: Sparkle.** Users on v1.3.0+ never see the GitHub-API poll again.

## Internal cleanup

- `Usage4Claude/Services/UpdateChecker.swift` — deleted. ~290 lines.
- `DataRefreshManager.scheduleDailyUpdateCheck` / `checkForUpdatesInBackground` / `checkForUpdatesManually` — deleted. The daily-update timer ID is gone; `dailyUpdateCheckScheduled` flag and `lastUpdateCheckTime` field are gone. `DataRefreshManager` is now strictly about Claude usage data.
- `MenuBarManager.checkForUpdates` — now a one-liner that calls `appDelegate.updaterController.checkForUpdates(self)`.
- `SemverCompare` — kept (still has tests and could be useful for future internal version logic, even though Sparkle handles version comparison itself).

## What's still on the backlog

- `MenuBarManager` still has `hasAvailableUpdate` / `latestVersion` / `acknowledgedVersion` / `shouldShowUpdateBadge` plus the rainbow-text notification flow — all dormant since nothing ever sets `hasAvailableUpdate = true`. Cleanup deferred to a follow-up to keep this commit's diff focused on the Sparkle integration. The dead UI doesn't crash; it just never fires.
- `UserSettings.simulateUpdateAvailable` debug flag — also dead; the badge it would have triggered no longer has a feed source. Will be removed alongside the badge cleanup.
- macOS Widget / Live Activity remains the last unimplemented P3 backlog item.

## Next steps for shipping

1. Build the v1.3.0 DMG: `./scripts/build.sh`. The script will notarize, staple, and print the `<enclosure>` line.
2. Update `appcast.xml`: prepend a new `<item>` for v1.3.0 with the printed enclosure. (Older items can stay for archaeology — Sparkle just looks at the newest version.)
3. Commit + push `appcast.xml` so the raw GitHub URL serves it.
4. Create the GitHub Release: `gh release create v1.3.0 build/.../U4Claude-v1.3.0.dmg --title "U4Claude v1.3.0" --notes-file docs/RELEASES/v1.3.0.md`.
5. Wait for v1.2.0 users to auto-prompt; they'll be the last cohort to use the legacy GitHub-API check.
