# U4Claude v1.6.1

Three small upstream backports — bug fixes that apply to our fork too.

## What changed

- **Refresh on system wake** — `DataRefreshManager` now subscribes to `NSWorkspace.didWakeNotification` and triggers a fetch ~3 seconds after the Mac wakes. Previously, smart-mode's idleLong tier could leave you sitting on stale data for up to 10 minutes after wake while the timer caught up. Backported from upstream commit [`de671c6`](https://github.com/f-is-h/Usage4Claude/commit/de671c6).

- **Idle-mode timer restart on user action** — When you opened the popover or hit manual refresh from a smart-mode idle tier (3/5/10 min), the in-memory mode flipped to `.active` but the *timer* kept running on the old longer interval. Now the timer is explicitly restarted on the idle→active transition, so the next tick fires at 1 min as expected. Same upstream commit.

- **5-hour and 7-day always shown in smart display mode** — `getActiveDisplayTypes` previously skipped them when the API returned no data. Every Claude account is subject to both limits, so they're now unconditional in smart mode. Backported from [`fffff55`](https://github.com/f-is-h/Usage4Claude/commit/fffff55).

- **7-day limit shows for new accounts** — `UsageResponse.toUsageData()` now emits a 0% placeholder for the 7-day limit when the API returns a missing field or `(utilization=0, resets_at=null)`. Brand-new accounts see the row from day one instead of an empty space. `addAccount` also posts `.accountChanged` after the first account add so `DataRefreshManager` picks it up immediately. Backported from [`1192f35`](https://github.com/f-is-h/Usage4Claude/commit/1192f35).
